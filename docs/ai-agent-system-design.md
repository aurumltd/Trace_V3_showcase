# Trace AI 智能体系统设计

本文是 Trace 产品案例的技术产品附录，说明智能体循环、工具、记忆、RAG、输出契约、失败回退和评估方式。它明确区分**当前测试版**与**后续设计**。

## 1. 产品目标与边界

> 基于真实活动、Calendar、Reminders 和用户修正，形成可解释的工作回放，判断计划推进状态，并提出有限的下一步建议。

智能体不负责：

- 替用户维护完整任务系统
- 在证据不足时猜测用户意图
- 静默覆盖用户修改
- 以聊天界面取代主要工作流
- 在模型不可用时阻塞产品

## 2. 能力状态

| 能力层 | 测试版状态 | 说明 |
|---|---|---|
| 活动感知 | 已实现 | macOS 活动窗口、时长和系统状态 |
| 工作理解 | 已实现 | 规则、语义字段、上下文键和学习规则归并 |
| 上下文工具 | 已实现 | Calendar、Reminders、活动历史 |
| 计划智能体 | 已实现测试版 | 确定性计划 + 可选本地模型结构化计划 |
| 执行监控 | 已实现测试版 | 计划块与实际工作块对照 |
| 复盘 | 已实现测试版 | 指标、偏差、结构化本地模型总结与回退 |
| 用户记忆 | 已实现基础版 | 可查看和清理的学习规则 |
| 向量检索/RAG | 已设计，未实现 | 计划在本地索引历史证据 |
| 智能体评估集 | 指标已定义 | 需要标注样本和回归基准 |

## 3. 智能体循环

```mermaid
flowchart LR
  A[感知<br/>活动/日历/提醒] --> B[理解<br/>工作块]
  B --> C[计划<br/>优先级与时间块]
  C --> D[行动<br/>展示或可选写回]
  D --> E[观察<br/>计划与实际对照]
  E --> F[反思<br/>Did / Drift / Next]
  F --> G[用户修正]
  G --> H[本地学习规则]
  H --> B
```

该循环体现 Agent 的关键特征：持续读取环境、使用工具、生成结构化行动、观察结果，并依据用户反馈调整后续判断。它不是一次性提示词调用。

## 4. 组件与责任

| 组件 | 输入 | 输出 | 当前机制 | 风险控制 |
|---|---|---|---|---|
| 活动感知 | 应用、窗口、时间、心跳 | 原始活动 | macOS 本地捕获 | 权限状态、睡眠/唤醒处理 |
| 工作理解 | 原始活动、规则、修正 | 工作块、分类、上下文键 | 确定性聚合与规则 | 可编辑、可解绑 |
| 上下文构建 | Calendar、Reminders、活动历史 | 日程、待办、警告 | 原生工具读取与缓存 | 超时、缓存、健康状态 |
| 计划智能体 | 空闲时间、待办、工作动量 | 计划块与理由 | 规则排程 + 本地模型可选增强 | JSON 解析、确定性回退 |
| 执行监控 | 计划块、实际工作块 | 执行状态 | 时间与语义匹配 | 显示依据、允许手动关联 |
| 复盘 | 摘要指标、偏差、警告 | Did / Drift / Next | 模板 + 本地模型 | 缓存、结构化回退 |

## 5. 工具使用

| 工具 | 智能体用途 | 失败方式 | 回退 |
|---|---|---|---|
| 活动追踪 | 获取真实行为 | 权限关闭、进程中断 | 清晰状态，不生成虚假记录 |
| Calendar | 读取时间约束、可选写回 | 拒绝权限、超时、冲突 | 缓存、重试、保护用户编辑 |
| Reminders | 读取用户计划意图 | 权限或列表不可用 | 使用已有上下文并显示警告 |
| 本地活动历史 | 判断工作动量与实际执行 | 数据不足 | 降低置信度 |
| 学习规则 | 复用用户纠错 | 错误规则累积 | 可查看、删除和重置 |
| 本地模型 | 生成结构化计划和复盘 | 未安装、超时、格式错误 | 确定性计划和模板摘要 |
| 本地检索/RAG | 检索历史证据 | **尚未实现** | 使用显式上下文和规则 |

## 6. 记忆设计

Trace 不使用开放式“聊天记忆”，而使用与产品任务直接相关、可解释的本地状态。

### 短期上下文

- 当日活动和当前工作块
- 当日 Calendar 事件
- 未完成 Reminders
- 当前计划建议和执行状态
- 当前周期复盘缓存

### 长期本地状态

- 用户修正后的分类和上下文键
- 手动绑定/解绑的 Calendar 与 Reminders
- 忽略应用
- 分类规则
- 模型与数据源设置

### 记忆写入原则

1. 只从显式修正或稳定设置中学习。
2. 不把一次模型推断直接升级为长期事实。
3. 用户可以看到、重置或删除规则。
4. 相互冲突的规则应请求确认，而不是静默覆盖。

## 7. RAG / 上下文锚定设计

**状态：已设计，尚未在当前测试版实现向量索引。**

RAG 的目标不是通用问答，而是为工作理解、计划和复盘提供用户自己的证据。

| 检索对象 | 使用方 | 目的 |
|---|---|---|
| 相似历史工作块 | 工作理解、执行监控 | 识别项目和工作语义 |
| Reminders | 计划智能体 | 锚定任务来源和优先级 |
| Calendar 事件 | 上下文构建、计划智能体 | 识别固定约束 |
| 学习规则 | 工作理解 | 复用用户确认过的语义 |
| 近期复盘摘要 | 复盘 | 识别重复偏差和长期主线 |

每条检索证据至少包含：

```json
{
  "sourceType": "work_block",
  "sourceId": "local-id",
  "title": "Trace case study",
  "matchedReason": "same project and window-title tokens",
  "score": 0.82,
  "timestamp": "local timestamp"
}
```

### RAG 质量门槛

- 先过滤权限范围和时间范围，再做相似度排序
- 限制 top-k，避免无差别塞入原始记录
- 重要建议显示来源和匹配原因
- 低于阈值时不引用
- 检索失败时回退到显式 Calendar/Reminders 和规则

## 8. 计划智能体输出契约

计划输出必须结构化，UI 不直接依赖一段自由文本。

```json
{
  "summary": "先延续当前主线，再处理低压力事项。",
  "blocks": [
    {
      "title": "完善产品案例",
      "startTime": "14:00",
      "endTime": "15:00",
      "sourceReminder": "更新作品集",
      "confidence": "high",
      "rationale": "今天已在同一项目投入时间，继续推进可减少切换。",
      "nextAction": "先完成问题定义与评估章节。",
      "prepHint": "打开案例文档和实现状态表。",
      "energy": "high_focus",
      "priorityReason": "与当前交付主线直接相关。",
      "evidence": []
    }
  ]
}
```

### 输出验证

- 时间必须可解析且不与固定日程冲突
- 时长必须位于允许范围
- 来源任务必须来自已读取上下文或明确标记为系统建议
- 置信度与证据强度一致
- 解析失败时丢弃模型结果并启用确定性计划

## 9. 执行状态

| 状态 | 判断 |
|---|---|
| 已完成 | 实际匹配时长接近计划时长 |
| 推进中 | 有明显相关工作但尚未达到计划时长 |
| 已开始 | 只有少量相关工作 |
| 待开始 | 尚无相关工作 |
| 明显偏移 | 计划时间内出现明显无关工作 |
| 无法判断 | 数据或证据不足 |

“无法判断”是必要状态，它避免系统把未知包装成偏差。

## 10. 失败模式与安全回退

| 失败模式 | 用户风险 | 产品响应 |
|---|---|---|
| 错误归并不同任务 | 回放失真 | 拆分/编辑入口，规则学习 |
| 错误关联计划 | 产生错误复盘 | 展示来源，支持绑定/解绑 |
| 模型输出泛化建议 | 失去信任 | 要求具体下一步和依据 |
| 模型输出格式错误 | 页面不可用 | Schema 校验与确定性回退 |
| 权限或工具失败 | 数据缺失 | 明确警告，不伪造完整性 |
| 过度记忆 | 错误持续放大 | 仅学习显式修正，可重置 |
| RAG 检索错误 | 错误证据增强判断 | 阈值、来源展示、无证据回退 |

## 11. 评估设计

### 离线数据集

建立匿名或合成测试场景，覆盖：

- 同一任务跨多个应用
- 相似窗口标题属于不同项目
- Calendar 与实际活动冲突
- Reminder 被部分推进
- 用户手动解绑后不应再次自动绑定
- 本地模型不可用或返回无效 JSON
- 睡眠/唤醒和长时间空闲
- 检索结果为空、冲突或低相关

### 核心指标

| 层级 | 指标 |
|---|---|
| 理解 | 工作块边界 F1、分类准确率、错误归并率 |
| 关联 | Calendar/Reminders 匹配精确率、错误自动绑定率 |
| 计划 | 建议采纳率、执行匹配率、建议修改率 |
| 信任 | 用户修正率、重复修正下降率、无依据建议率 |
| RAG | Recall@k、证据精确率、引用支持率 |
| 可靠性 | 工具失败率、回退成功率、模型解析失败率 |

### 上线门槛

在扩展主动能力前，至少需要证明：

1. 核心流程在模型关闭时仍可完成。
2. 自动关联以精确率优先，不以覆盖率优先。
3. 用户修正能减少重复错误。
4. 低置信度场景能正确进入“无法判断”或确认流程。
5. RAG 引入后有依据建议增加、错误引用不增加。

## 12. 产品原则

Trace 的 Agent 应该：

- 使用工具，但不静默接管工具
- 使用记忆，但只记住可解释事实
- 使用生成模型，但保留确定性回退
- 使用 RAG，但不把相似度当成事实
- 给出建议，也给出依据和置信度
- 在不知道时明确说不知道

其价值不是替用户做所有决定，而是让用户更快看清事实、更少重复修正、更容易进入下一步行动。
