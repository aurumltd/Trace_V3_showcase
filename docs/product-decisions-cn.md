# Trace 产品决策记录

本文只记录关键取舍，避免重复完整案例。问题背景、实现状态和评估框架见 [产品案例](portfolio-case-study-cn.md)。

## 决策 1：不做新的任务系统

- **备选方案：** 内建任务、日历、团队协作和完整计划流程
- **选择：** 读取 Calendar 和 Reminders，只补足事实、理解与下一步建议
- **原因：** 降低迁移成本，把资源集中在现有工具没有解决的工作回放问题
- **代价：** 上下文质量受系统权限和外部数据完整度影响

## 决策 2：先形成工作块，再生成建议

- **备选方案：** 直接把原始活动交给 LLM 总结
- **选择：** 先用确定性逻辑聚合可审查的工作块，再进入计划和复盘
- **原因：** 原始事件噪音大，无法稳定支持解释、纠错和评估
- **代价：** 需要维护归并规则、边界测试和纠错机制

## 决策 3：智能体嵌入主流程，不以聊天为主界面

- **备选方案：** 做一个“效率教练”聊天窗口
- **选择：** 把感知、计划、观察和复盘放入 Today、Timeline 和 Review
- **原因：** 用户需要持续上下文和可操作状态，而不是一次性对话
- **代价：** 产品必须为每个页面定义清晰职责和结构化输出

## 决策 4：把纠错作为信任与学习闭环

- **备选方案：** 只允许修改当前显示结果
- **选择：** 修正当前记录，并保存可查看、可重置的本地规则
- **原因：** 同类错误不应要求用户反复修正；用户必须控制系统如何学习
- **代价：** 需要处理规则冲突、解绑和错误规则累积

## 决策 5：本地模型是增强，不是单点依赖

- **备选方案：** 所有计划和复盘都依赖 LLM
- **选择：** 使用本地模型生成结构化结果，同时保留确定性计划和模板回退
- **原因：** 模型可能未安装、超时或返回无效格式，核心价值不能因此中断
- **代价：** 需要维护两条路径并保证结果含义一致

## 决策 6：RAG 作为下一阶段证据层，而不是当前实现卖点

- **备选方案：** 在文档中把“读取历史数据”泛称为 RAG
- **选择：** 明确当前是显式上下文和规则；向量检索、top-k 证据与引用属于后续设计
- **原因：** 技术术语必须对应真实机制，避免把路线图包装成已交付能力
- **代价：** 当前个性化主要依赖规则和有限历史上下文

## 决策 7：自动关联以精确率优先

- **备选方案：** 尽量把每个工作块都匹配到计划
- **选择：** 弱证据时保持“无法判断”，并允许手动绑定或解绑
- **原因：** 错误关联会污染计划对照和长期复盘，比缺少关联更伤害信任
- **代价：** 初期覆盖率较低，需要用户确认部分结果

## 决策 8：明确区分已实现、已设计和待验证

- **已实现：** 测试版代码和可运行流程
- **已设计：** 有系统方案但尚未完成的 RAG 与评估能力
- **待验证：** 用户价值、长期授权、留存与效率影响
- **原因：** AI 产品作品集的可信度来自证据边界，而不是功能名词数量
