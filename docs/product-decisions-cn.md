# Trace 产品决策

这份文档用于说明 Trace 背后的关键产品判断，重点展示产品取舍、AI 产品设计深度和高级产品经理能力。

## 决策 1：不再做一个新的任务系统

**问题：** 用户已经有日历、提醒事项、笔记或轻量任务习惯。要求用户迁移到一个新系统，会显著增加使用门槛。

**决策：** Trace 读取用户已有计划上下文，而不是替代它。

**意义：** 这让 Trace 成为理解层和辅助层，而不是另一个需要维护的任务数据库。

## 决策 2：从事实出发，而不是从主观填报出发

**问题：** 手动时间记录依赖记忆和自律，越忙越容易失败。

**决策：** Trace 从真实活动捕获开始，再把活动归并成可理解的工作块。

**意义：** 产品先建立事实基础，再让 AI 总结、判断和建议。

## 决策 3：用 AI 理解工作，而不是只总结文字

**问题：** 泛泛的 AI 总结可能听起来有用，但并不能判断计划是否被推进。

**决策：** Trace 把 AI 能力拆成工作理解、计划生成、执行监控和复盘总结。

**意义：** 智能体有明确产品职责：把噪音行为转化成和计划相关的判断。

## 决策 4：把修正能力做成核心闭环

**问题：** AI 一定会误判窗口标题、项目上下文和用户意图。

**决策：** 用户可以修正标题、分类、上下文键、时间范围、日历关联和提醒事项关联。

**意义：** 修正不是补充功能，而是信任机制和学习机制。用户修正会转化为本地学习规则。

## 决策 5：用上下文和检索增强支撑建议

**问题：** 如果不基于用户自己的工作证据，智能体建议很容易变成泛泛建议。

**决策：** 使用本地检索增强，检索相关工作块、提醒事项、日历约束、学习规则和近期复盘摘要。

**意义：** 建议应该能解释依据，而不是黑箱输出。

## 决策 6：低置信度时回退，而不是假装确定

**问题：** 过度自信的 AI 计划会快速损害用户信任。

**决策：** 当证据不足或工具失败时，Trace 回退到显式规则、缓存上下文和可编辑建议。

**意义：** 一个可靠的 AI 产品应该在不确定时请求修正，而不是强行自动化。

## 决策 7：坚持本地优先

**问题：** 工作活动数据高度敏感，用户不一定愿意把原始行为记录发送到外部系统。

**决策：** 优先本地存储、本地规则、本地上下文和可用时的本地 AI 总结。

**意义：** 隐私不是单纯技术要求，而是产品信任模型和采用路径的一部分。

## 决策 8：区分今日页、时间线页和复盘页的职责

**问题：** 很多生产力产品把仪表盘、日志和复盘混在同一个页面，导致信息任务不清。

**决策：** 今日页回答“现在怎么办”，时间线页回答“具体发生了什么”，复盘页回答“我应该调整什么模式”。

**意义：** 每个页面都有明确用户任务和智能体职责。

## 决策 9：明确区分已实现能力和路线图能力

**问题：** AI 产品作品集容易把路线图概念写得像已经上线。

**决策：** README 明确区分已实现的测试版能力和设计中的路线图能力。

**意义：** 清晰的状态边界能增强高级产品评审中的可信度。
